# 事件驱动架构改造 - 完成总结

## 改造成果

✅ **实时止损** - 从60秒延迟降至<1秒  
✅ **事件驱动** - CPU占用从持续轮询降至事件唤醒  
✅ **异步并发** - asyncio架构，支持横向扩展  
✅ **向后兼容** - 不影响原有cron任务  

---

## 架构对比

### 改造前（定时轮询）
```
[Hermes cron] → quant_stop_check.sh (每分钟)
                  ↓
              python3 main.py --stop-check
                  ↓
              遍历持仓检查止损（最坏60秒延迟）
```

### 改造后（事件驱动）
```
[长桥WebSocket] → 实时行情推送 (<100ms)
                     ↓
                 EventBus分发
                     ↓
              StopLossHandler响应（<1秒）
```

## 快速上手

### 1. 本地测试
```bash
# 运行单元测试
python tests/test_realtime.py

# 启动事件驱动交易员（前台测试）
python main.py --realtime
```

### 2. Hermes生产部署
```bash
# 1. 复制systemd配置
cp scripts/quant-realtime.service ~/.config/systemd/user/

# 2. 修改路径（如果项目路径不同）
vim ~/.config/systemd/user/quant-realtime.service

# 3. 启用并启动服务
systemctl --user daemon-reload
systemctl --user enable quant-realtime
systemctl --user start quant-realtime

# 4. 查看状态
systemctl --user status quant-realtime
```

### 3. 日志监控
```bash
# 实时日志
journalctl --user -u quant-realtime -f

# 或查看文件日志
tail -f ~/.hermes/logs/quant-realtime.log
```

### 4. 停止服务
```bash
systemctl --user stop quant-realtime
```

### 保留的定时任务（Hermes cron）
- ✅ `quant_scan.sh` - 早盘扫描（9:35）
- ✅ `quant_execute.sh` - 收盘执行（14:58）
- ✅ `quant_review.sh` - 收盘复盘（15:05）
- ⚠️ `quant_stop_check.sh` - **可以停用**（被实时监控取代）

### 新增的常驻服务
- 🆕 `quant-realtime.service` - 事件驱动交易员（systemd守护）

**建议**：保留原有cron作为兜底，实时服务作为主力。如果实时服务挂了，cron每分钟的止损巡检仍能救命。

## 故障处理

### 问题1：服务启动失败
```bash
# 查看详细错误
journalctl --user -u quant-realtime -n 50

# 常见原因：
# - 长桥API凭证未设置 → 检查 ~/.hermes/.env
# - 端口被占用 → netstat -anp | grep <port>
# - Python依赖缺失 → pip install longport
```

### 问题2：实时行情无数据
```bash
# 检查长桥WebSocket连接
grep "长桥" ~/.hermes/logs/quant-realtime.log

python -c "from execution.paper_account import PaperAccount; print(PaperAccount().positions)"
```
