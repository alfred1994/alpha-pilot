# 模拟盘可靠性约定

更新日期：2026-07-16
执行者：Codex

## 账户与成交

- 系统仅支持 `BROKER_MODE=paper` 的模拟盘运行。
- 买入和卖出会拒绝零值、负值、非有限数以及不符合整手约束的输入。
- 一笔模拟成交会在同一 SQLite 事务内同时写入账户快照、持仓投影和交易记录；SQLite 是唯一事实来源。
- `data/paper_account.json` 仅是账户备份，采用临时文件加原子替换写入。数据库提交成功后即使备份失败，也不会回滚已经完成的成交。

## TradePlan 执行

- 每个 TradePlan 根据其稳定内容生成 `plan_id`，数据库会记录领取和完成状态。
- 同一计划不会因重试、进程重启或并发调用而重复下单。
- 自动执行只接受当前交易日、未超过 `deadline` 的计划；历史计划只能在显式回放场景中执行。

## 回测口径

- 信号使用前一交易日已收盘数据，成交发生在下一交易日。
- A 股普通股票默认执行 T+1 卖出限制。
- 佣金、最低佣金和卖出印花税由持仓管理器统一计算，并计入现金和交易记录。
- 回测结束使用最后可得价格估值，不通过强制平仓绕过交收限制。

## Web 看板

- `GET /positions` 只用实时价格计算本次响应，不修改账户、持仓或数据库。
- `GET /trades` 只读取交易记录，不在查询时修复或回填历史盈亏。
- 账户读取失败会标识为不可用，不再伪造默认的一百万元账户。

## 验证与回滚

- GitHub Actions 的 `Test AlphaPilot` 工作流在干净的 Python 3.12 环境中安装依赖、执行依赖检查、编译和回归测试。
- 回滚代码时应一并回滚相应数据库变更；新增表 `trade_plan_executions` 保留历史执行审计，不包含账户密钥或凭据。
- 发生账户数据异常时，先停止自动执行，再备份 SQLite 文件后使用交易记录和账户快照核对，不要手工修改公开看板接口来“修复”数据。
